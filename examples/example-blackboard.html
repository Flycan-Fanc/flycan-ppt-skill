核心问题 02 / 05
Q

为什么我们需要 React？

直接操作 DOM 的问题：

• 命令式操作：每一步都得告诉浏览器做什么
• 状态与视图不同步：手动维护 DOM 状态
• 难以管理复杂交互：事件、数据、UI 交织
React 的回答：让 UI 成为状态的函数

推导过程 03 / 05
核心概念推导

从问题到方案

01
声明式 UI
描述"想要什么"，而不是"怎么做"。UI = f(state) 的核心含义。
02
组件化
将 UI 拆分为独立、可复用的组件。每个组件管理自己的状态。
03
单向数据流
数据从上往下流动，状态变化触发重新渲染。
这三个概念构成了 React 的基石：声明式 + 组件化 + 单向数据流

对比分析 04 / 05
编程范式

声明式 vs 命令式

命令式（直接操作 DOM）
// 每一步都要手动操作
const el = document.getElementById('root');
el.innerHTML = '';
const h1 = document.createElement('h1');
h1.textContent = count;
el.appendChild(h1);
痛点：状态越多，代码越复杂
声明式（React）
// 只关心"要显示什么"
function App() {
  return <h1>{count}</h1>;
}
优势：关注结果，而非过程
声明式不是 React 的专利，它是人类解决问题最自然的方式
